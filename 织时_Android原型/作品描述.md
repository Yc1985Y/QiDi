# 织时：将校园信息碎片整合为专属时间线的智能助手

## 一、作品整体定位

`织时` 是一个面向高校场景的 Android 校园智能体应用。它的核心目标不是做一个普通的日历工具，也不是做一个只会“看图说话”的识别 Demo，而是把校园里分散、非结构化、来源复杂的信息碎片，转化为一条可以确认、可以提醒、可以回看、可以导出的个人时间线。

这款作品所解决的问题非常具体：大学生日常会接触大量海报、群通知、截图、网页公告、教务文本和口头指令，但这些信息往往分散在不同渠道中，时间、地点、事件类型和提醒需求并不统一。传统做法需要用户自己阅读、提炼、记忆、再手动录入日历，认知摩擦很高，也非常容易漏看和遗忘。`织时` 的价值就在于把这一步前置为“系统理解”，让用户只做最后确认。

从产品范式上看，`织时` 已经不是传统的“输入表单型 App”，而是一个围绕“输入碎片 -> 理解意图 -> 结构化建议 -> 用户确认 -> 系统执行 -> 时间线沉淀”构建的校园智能助手。

---

## 二、当前作品的整体形态

目前作品已经形成一个可以运行、可以安装、可以演示的 Android 原型，整体是标准的移动端 App 形态，而不是单页 Demo。

当前应用由以下几部分组成：

1. 启动页
2. 首页
3. 时间线页
4. 我的页
5. 底部导航栏
6. 加载遮罩与错误反馈层

桌面图标名称已经统一为：`织时`

应用内部的作品全称为：`织时：将校园信息碎片整合为专属时间线的智能助手`

底部导航是三栏结构：

- 首页
- 时间线
- 我的

这意味着作品已经从“单次识别流程”升级为“可持续使用的产品结构”，每个页面都有独立职责，页面之间可以顺滑切换，并通过状态共享形成完整闭环。

## 三、设计总原则

当前作品的 UI 和交互不是按照“工程调试面板”的思路设计的，而是按照“正常可展示、可理解、可答辩的产品界面”来重构的。整体设计遵循以下原则：

### 1. Today-First

首页不先展示繁杂配置，而是先让用户感知“今天、此刻、这次导入”的价值，强调“当下要处理什么”。

### 2. 信息降噪

页面不鼓励用户自己去手动拆分信息，而是尽量把输入、识别、确认和沉淀后的结果可视化，减少阅读负担。

### 3. 卡片化表达

界面以圆角卡片、信息分层和便当盒式 Bento 区块为核心，避免过密表格和生硬列表。

### 4. 先解释，再执行

作品不会在识别后直接执行系统动作，而是通过风险判断和确认卡片把控制权留给用户。

### 5. 正常 App 化

作品必须像一个真实可用产品，而不是竞赛答辩用的临时页面，因此加入了底部导航、计划沉淀页、个人控制页、导出能力和提醒策略展示。

## 五、启动页设计

启动页是用户进入 `织时` 后看到的第一层界面，当前已经不是早期那种多页 PPT 式介绍，而是一个较为完整的沉浸式品牌启动页。

### 1. 页面整体视觉

启动页使用纵向渐变背景，颜色从深海蓝过渡到浅白蓝，形成一种“从复杂信息中抽丝剥茧、逐步澄清”的视觉感受。这个背景与作品名称“织时”中的“编织、整理、串联”语义保持一致。

### 2. 页面上半部分

页面中上部放置品牌主标题与视觉中心区域：

- 顶部小字号品牌名：`织时`
- 中心品牌标签：`Campus Timeline Agent`
- 大标题：`将校园信息碎片整合为专属时间线`

这个区域负责建立用户对作品定位的第一印象，让用户一进入应用就知道这不是一个简单记事本，而是一个可以整理校园碎片信息的智能体。

### 3. 页面中部说明

中部用一段简洁说明介绍产品能力：

- 织时会把海报、群通知、截图与口头指令整理成可确认、可提醒、可回看的校园时间线

这段文案的作用是把作品核心链路用一句话讲明白。

### 4. 页面下部操作

底部保留一个主按钮：

- `开启智能体`

按钮为深色圆角主按钮，点击后进入应用主界面。

### 5. 启动页的作用

启动页的职责不是承载复杂操作，而是完成三件事：

- 建立品牌感
- 建立作品定位
- 平滑进入主流程

## 六、首页设计与功能说明

首页是整个作品最核心的页面，它既是输入入口，也是 AI 工作台，也是答辩时最重要的主演示页面。

当前首页从视觉上分为五个层次：

1. Today-First 顶部焦点卡
2. 意图输入舱
3. 场景捷径区
4. 动态工作台区
5. 快捷继续区

页面整体背景采用深蓝到浅白的纵向渐变，并叠加轻量网格纹理，形成“智能系统正在运行”的科技感，同时避免页面显得空。

### 1. 顶部焦点卡

首页最上方是一张大尺寸深色英雄卡片，也可以理解为“今日焦点卡”。

卡片内包含：

- 状态胶囊标签
- 品牌小标题 `织时`
- 主标题 `把校园信息碎片整合成一条属于你的时间线`
- 当前摘要文本
- 一行双信息小卡

#### 状态胶囊

状态胶囊会根据当前运行状态变化：

- `Today First`
- `织时正在整理碎片`
- `正在听你说话`

这让用户随时知道当前系统处于什么状态。

#### 主标题

主标题直接表达首页价值，而不是写“欢迎使用”等无效文本。

#### 摘要文本

摘要文本会优先展示当前解析结果或状态信息。如果当前还没有新的识别结果，就显示默认说明，告诉用户只要把海报、截图或一句话交给系统，系统就会帮他先理解，再进入确认。

#### 底部双信息

双信息卡展示：

- 当前入口
- 运行方式

例如：

- 当前入口：分享文本、相册导入、拍照识别等
- 运行方式：识别 -> 建议 -> 确认

这一部分的意义在于把产品逻辑明示出来，尤其适合答辩展示。

### 2. 意图输入舱

这是首页第二个核心模块，位于顶部焦点卡下方，是一个白色圆角输入卡片。

该区域包含：

- 模块标题 `意图输入舱`
- 一段提示说明
- 一块大输入框
- 一行能力标签 Chips
- 两行操作按钮

#### 说明文字

说明文案会明确告诉用户：

- 可以扔一张海报
- 可以扔一段群通知
- 可以扔一个截图
- 也可以直接说“帮我记下来”

这实际上是在建立用户心智：不要求标准输入，允许碎片化输入。

#### 输入框

输入框支持用户直接输入文本型通知，例如：

`周五晚七点图书馆报告厅有讲座，帮我加入时间线`

这块输入框不是传统的表单字段，而是面向自然语言与片段信息的意图输入区。

#### 能力标签

输入框下方有一排轻量胶囊标签：

- 📷 拍海报
- 🖼️ 选截图
- 🎤 语音指令
- ✂️ 粘贴文本

这些标签不是主要按钮，而是对系统能力的提示与教育。

#### 第一行主按钮

第一行两个主按钮：

- `发送解析`
- `拍照识别` 或 `打开相机`

这里有一个已经实现的性能优化点：相机不是默认持续占用资源，而是在用户真正需要时才打开。若首页当前处于相机待机状态，按钮会显示“打开相机”；一旦进入预览态，按钮才真正承担拍照识别动作。

#### 第二行快捷按钮

第二行三个快捷按钮：

- 相册导入
- 粘贴文本
- 语音输入

这一层让首页具备多入口能力，支持从不同素材形态进入同一条 AI 流程。

### 3. 场景捷径区

意图输入舱下方是场景捷径区，用两列卡片的方式展示预设场景。

当前场景包括：

- 海报入日历
- 地点去导航
- 长文本播报
- 群通知转日程

每张场景卡片内部包含：

- 图标
- 场景标题
- 一小段预设提示词摘要
- `套用此场景` 按钮

这个区域的作用有两点：

- 降低首次使用门槛
- 让答辩时的场景演示更直接

### 4. 动态工作台区

这是首页最重要的 AI 展示区域，也是当前 UI 重构中最具“产品完成度”的部分。它位于场景捷径区下方，以 Bento 分块方式呈现。

动态工作台根据屏幕宽度自适应布局：

- 小屏设备下采用上下堆叠
- 大屏或宽屏下采用左右并排

动态工作台主要由两大部分组成：

#### 4.1 输入源画面卡

左侧或上方卡片叫做 `输入源画面`。

它有三种状态：

1. 相机在线预览
2. 相机待机未打开
3. 静态导入占位

如果设备支持实时预览且用户已打开相机，这里显示真实 CameraX 预览画面。

如果相机暂时未打开，则展示一个深色占位卡，提示：

- 相机已收纳到待机状态
- 需要拍海报时再点击打开
- 相机不会在首页空转，切页会更轻

这个设计不仅解决性能问题，也使界面表达更成熟。

如果当前主要使用静态导入方式，则这里会展示静态导入说明占位。

卡片底部还会显示当前模式标签：

- 相机
- 在线预览 / 待机 / 静态导入

#### 4.2 结果工作台卡

右侧或下方是 `动态工作台` 主卡片。

它是首页最核心的“解释层”，当前包含以下内容：

- 模块标题
- 一段工作流说明
- 两张信息卡
- 工作流阶段卡
- 信心指数卡
- 结构化提取结果卡
- 确认写入卡

##### 最近导入卡

展示本次素材从哪里进入，比如：

- 相册导入
- 分享图片
- 分享文本
- 剪贴板
- 手动文本
- 相机拍照

##### 识别摘要卡

展示本次系统对内容的摘要理解。这块内容不是原始 OCR 全量输出，而是系统准备给用户看的摘要结果。

##### 工作流阶段卡

这一块把本次导入拆解为三个阶段：

- 导入
- 抽取
- 确认

每个阶段都对应一个小卡片，卡片内会显示当前阶段说明，例如：

- 导入：等待素材 / 已导入
- 抽取：字段已生成 / 待结构化
- 确认：待你确认 / 等待建议

这个设计非常适合答辩，因为它直接把系统的内部逻辑可视化了。

##### 信心指数卡

这是当前首页设计中的重要工程亮点。

该卡片包含：

- 标题 `信心指数`
- 百分比数字
- 一条可视化进度条
- 一段风险解释文案

根据当前置信度，系统会给出不同提示：

- 高置信度：可直接进入确认写入
- 中置信度：建议用户快速核对
- 低置信度：建议补充信息或重新导入

这解决了 AI 系统最容易被质疑的一个问题：识别不确定时怎么办。

##### 结构化提取结果卡

如果模型已经返回结构化意图，系统会展示一张字段结果卡，内容包括：

- 标题
- 时间
- 地点
- 动作
- 补充说明

这里显示的是“系统准备执行什么”，而不是抽象自然语言。也就是说，首页已经把 AI 输出从普通文本进一步转换为可执行字段。

动作当前可能包括：

- 加入时间线
- 生成导航
- 继续追问
- 语音摘要

##### 确认写入卡

当风险引擎判断当前动作需要确认时，会在工作台底部显示确认卡。

确认卡包含：

- 标题 `确认写入系统`
- 模型建议话术
- 标题/时间/地点字段预览
- 风险等级与执行模式
- 两个按钮：`确认写入`、`取消`

这是“先解释，再执行”原则最具体的可视化实现。

### 5. 快捷继续区

首页最下方还有一个 `快捷继续` 卡片，里面有三颗按钮：

- 语音
- 相册
- 粘贴

这个区域的作用是让用户在完成一次识别后，可以不回到页面顶部，直接继续导入下一条碎片信息。

同时这一卡片也用一句话强调了作品的核心能力：

- 重点不是单一拍照，而是让海报、截图、分享文本、语音和粘贴内容都能流入同一条 AI 时间线闭环

## 七、时间线页设计与功能说明

时间线页承接首页确认后的结果，是整个作品从“识别工具”升级到“时间管理产品”的关键页面。

它的职责不是只显示一条任务，而是把所有已经确认或待确认的安排组织成一个可浏览、可展开、可提醒、可导出的计划集合体。

当前时间线页从上到下分为以下模块：

1. 页面英雄卡
2. 两组摘要信息卡
3. 待确认卡
4. 日 / 周 / 月切换器
5. 月历卡
6. 周节奏条
7. 时间线详情区
8. 提醒策略卡
9. 导出卡
10. 最新摘要与导入指令回顾
11. 返回首页按钮

### 1. 页面英雄卡

顶部是一张深色英雄卡，标题为：

`把零散校园信息沉淀成可回看的专属计划流`

副说明强调计划页不再只是任务列表，而是按日、周、月把已确认安排编织成个人时间线。

这张卡在答辩中起到“重新定义计划页价值”的作用。

### 2. 摘要信息卡

页面接着展示两排双信息卡，主要展示：

- 今日安排
- 已确认
- 最近导入
- 当前状态

状态会根据系统运行情况动态变化，例如：

- 织线中
- 待你确认
- 已织入
- 还没有安排

这样一来，时间线页并不是死数据页面，而是和首页状态联动的。

### 3. 待确认卡

如果当前首页刚识别出一个需要确认的安排，时间线页也会直接展示确认卡，用户可以在这个页面继续完成确认操作，而不一定要退回首页。

这保证了跨页操作的连续性。

### 4. 日 / 周 / 月切换器

时间线页中央有一个胶囊式切换器，支持三种视图：

- 日视图
- 周视图
- 月视图

切换器采用大圆角、清晰选中态和深浅对比，符合移动端常规使用习惯。

### 5. 月历卡

月历卡是当前计划页中的主可视化模块之一。

月历卡包含：

- 当前月份标题
- 上下月切换按钮
- 星期标题行
- 月视图日期网格

每一个日期单元格都支持表达“该天是否有安排”：

- 若无安排，则为浅色空白格
- 若有安排，则显示淡色高亮和简短摘要
- 若当前选中，则显示深色高亮

如果一天只有一个安排，就显示该安排标题前几字。
如果一天有多个安排，就显示“X 项安排”。

这满足了用户“先看分布，再看细节”的习惯。

### 6. 周节奏条

月历卡下方是一个横向的 `本周节奏` 区域。

它会按一周七天展示简短日块，每一天会显示：

- 日期
- 当天安排数量
- 当前是否为选中日

这部分的作用是让时间线页不仅适合看整体，还适合看一周节奏和密度。

### 7. 时间线详情区

这是计划页最关键的内容区域，承担“真正把计划组织起来”的职责。

它根据视图模式不同，会呈现不同形式：

#### 日视图

如果选择某一天，系统会把该天所有安排按照时间顺序列出来，形成竖向时间线。

每一条时间线条目包含：

- 左侧时间线圆点和竖线
- 时间
- 状态
- 标题
- 来源
- 地点
- 摘要
- 提醒策略说明
- 系统动作说明

当前版本中，用户还可以点击任意一条安排，拉起底部详情弹层，查看这条安排的完整解释信息，包括：

- 状态
- 完整时间
- 地点
- 来源
- 提醒策略
- 系统动作说明
- 摘要文本

例如一条已确认日程会显示：

- 时间：19:00
- 状态：已确认 或 已加入日历
- 来源：相册导入 / 分享文本 / 示例数据
- 地点：图书馆报告厅
- 提醒：提前1天 / 提前1小时
- 行为说明：已纳入系统级时间线与本地提醒

这种布局相比简单列表更有“时间线”感，也更符合作品名称“织时”。

#### 周视图

如果选择周视图，系统会列出本周若干天的可展开日卡片。

每张卡片展示：

- 日期
- 当天安排数量
- 收起状态下的两条摘要
- 点击后展开全部安排

这让周视图更适合做中尺度浏览。

#### 月视图

## 十、完整功能链条说明

当前作品最核心的优势，不是做了很多零散功能，而是已经形成了一条较完整的功能闭环。

完整功能链条如下：

### 1. 多入口输入

用户可以通过以下方式将校园信息交给系统：

- 拍照识别
- 相册导入
- 系统分享图片
- 系统分享文本
- 剪贴板粘贴文本
- 手动输入文本
- 语音输入

这意味着作品不是把“拍照”当成唯一入口，而是兼容真实校园使用中的多种信息来源。

### 2. 输入归一化

无论素材来自相机、截图、分享还是文字，系统最终都会封装为统一的 `CampusNoticeInput` 输入对象，交给同一条处理链路。

这一步保证了后续 AI 理解逻辑的一致性。

### 3. OCR 与大模型理解

如果输入是图片：

- 系统先将图片转换为 base64
- 调用 vivo OCR 接口提取文本
- 再把 OCR 文本与用户指令一并交给大模型做结构化理解

如果输入是文本：

- 直接把用户文本与当前意图送入大模型解析

当前云端能力已经接入 vivo 竞赛提供接口：

- LLM 接口：`https://api-ai.vivo.com.cn/v1/chat/completions`
- OCR 接口：`https://api-ai.vivo.com.cn/ocr/general_recognition`
- 模型：`Volc-DeepSeek-V3.2`

### 4. 结构化意图输出

模型并不是返回一段自由文本，而是被系统提示词强制约束为结构化 JSON。

当前约定的动作类型包括：

- `create_event`
- `navigate`
- `tts_feedback`
- `clarification`
- `unknown`

返回的字段包括：

- action
- confidence
- payload.title
- payload.time
- payload.location
- payload.description
- payload.answer
- fallback_query

### 5. 响应清洗与意图融合

模型返回结果后，系统会先做一次响应清洗，包括：

- 去除脏文本与 markdown 包装
- 标准化动作名
- 标准化时间格式
- 规整电话号码
- 统一字段位置

随后系统会把模型置信度、字段完整性、质量因子融合成 `fusedConfidence`，形成更稳健的执行置信度，而不是直接迷信模型原始分数。

### 6. 风险引擎决策

系统不会因为模型返回了“create_event”就直接执行，而是会把结构化结果送入 `RiskPolicyEngine`。

当前策略是：

- 高风险动作阈值更高
- 中风险动作需要确认
- 低风险动作可以直接播报
- 缺字段或置信度不足时，进入追问或阻断

执行模式当前分为：

- `DIRECT_TTS`
- `REQUIRE_CONFIRMATION`
- `REQUIRE_CLARIFICATION`
- `BLOCKED`

### 7. 首页工作台展示

风险引擎给出结果后，首页会同步展示：

- 当前摘要
- 工作流阶段
- 信心指数
- 结构化字段
- 确认卡片

也就是说，系统不是把结果藏在后台，而是把判断过程透明展示给用户。

### 8. 用户确认

如果动作属于中风险或高风险，例如：

- 创建日历事件
- 打开地图导航

系统会展示确认卡，用户必须点击 `确认写入` 才会真正进入系统层动作。

这一步是人始终在回路中的关键体现。

### 9. 系统执行

用户确认后，系统会调用 Android 原生能力执行动作：

#### 创建日历事件

通过 `Intent.ACTION_INSERT` + `CalendarContract.Events` 打开系统日历插入界面，并预填：

- 标题
- 时间
- 地点
- 描述

#### 打开导航

通过 `google.navigation:` 或 `geo:` URI 拉起地图应用。

#### 低风险语音反馈

对低风险内容，系统支持直接用 TTS 做播报反馈。

### 10. 本地提醒接入

当一条安排被确认加入时间线后，系统会调用 `ReminderScheduler` 为其生成本地提醒。

当前默认提醒策略是：

- 提前1天
- 提前1小时

提醒通过 WorkManager 调度，在触发时由 `ReminderWorker` 发出系统通知。

通知点击后会自动打开应用并进入 `时间线` 页。

### 11. 时间线沉淀

被确认的安排不会只停留在首页，而是会被转写为 `AgendaCardData`，沉淀进计划集合体中。

每一条数据都会包含：

- 唯一 id
- 标题
- 摘要
- 时间
- 地点
- 状态
- 来源
- 动作类型
- 提醒策略

这些数据最终在时间线页按日、周、月组织展示。

### 12. 本地持久化与偏好记忆

当前版本已经接入本地 `DataStore`，会将以下内容保存在设备侧：

- 时间线中的日程集合
- 默认提醒提前量
- 是否保留提前一天提醒
- 是否保留分钟级提醒
- 高风险动作拦截偏好
- 低置信度静默处理偏好
- 地点解析与地图联动偏好

这意味着应用杀后台或重新启动后，时间线与核心偏好不会丢失，作品已经从“内存态 Demo”进一步走向“可持续使用产品”。

### 13. 导出输出

时间线页面支持将当前视图导出为：

- PDF
- JPG
- PNG

这样作品的输出结果不仅能看，还能带出应用。

## 十一、技术架构说明

当前作品采用的是“Android 客户端 + 云端 AI 理解 + 本地系统执行”的分层架构。

### 1. 输入层

主要负责接收多入口输入：

- 相机
- 相册
- 分享
- 剪贴板
- 语音
- 文本输入

### 2. 预处理层

负责：

- 图片 URI 读取
- 图片 base64 编码
- OCR 前置处理
- 文本整理

### 3. 智能理解层

通过 `VLMNetworkClient` 对接 vivo 云端模型：

- 先 OCR
- 再 LLM
- 最终输出结构化 JSON

### 4. 结构化控制层

通过 `ResponseInterpreter` 和 `VisualActionIntentSchema` 将模型输出转换为系统可执行对象。

### 5. 风险决策层

通过 `RiskPolicyEngine` 判断：

- 置信度是否足够
- 字段是否完整
- 是否允许直接播报
- 是否必须确认
- 是否应继续追问

### 6. 执行层

通过 `IntentDispatcher` 调用：

- 日历
- 地图
- 本地 TTS

### 7. 提醒层

通过 `ReminderScheduler + WorkManager + ReminderWorker` 实现系统级本地提醒。

### 8. 展示层

通过 Jetpack Compose 实现：

- 启动页
- 首页
- 时间线页
- 我的页
- 加载层
- 错误层
