# 初版展示汇报材料

## 1. 项目名称

**VisualSemanticAgent：移动端视觉-动作操作系统（Visual-to-Tool OS）原型**

一句话定义：

> 一个把现实世界中的视觉信息转成结构化动作意图，并安全调用手机系统能力的 Agent Middleware OS。

---

## 2. 项目定位

这个项目不是传统意义上的“识图 App”。

它更接近一个运行在手机上的 **AI 中间件层**：

- 上游接收真实世界输入
- 中间完成语义理解和风险判断
- 下游调用系统工具执行动作

我们要解决的问题不是“模型能不能识别图片”，而是：

> 手机如何从“看懂”现实世界，走到“安全地替用户完成动作”。

---

## 3. 参赛方向契合

项目与赛题“手机 AI 助手和手机系统 AI 体验”高度契合，原因在于：

- 使用手机相机、语音、日历、地图、短信、TTS 等原生能力
- 用大模型做多模态理解和动作路由
- 强调的是系统级交互体验，而不是单点问答
- 可以在 vivo 手机上形成真实的“视觉输入 -> 工具调用”体验闭环

这个项目的亮点不是一个页面设计，而是：

**AI 如何充当手机系统和现实世界之间的翻译层。**

---

## 4. 我们要解决的核心问题

现实世界中的大量信息是碎片化的：

- 海报上的时间和地点分散在不同区域
- 通知单上的地址不适合直接导航
- 设备铭牌上的电话和说明需要提炼后再沟通
- 菜单、资料、说明书太长，不适合直接 OCR 朗读

传统流程下，用户必须：

1. 看懂信息
2. 记住信息
3. 打开另一个 App
4. 手动输入
5. 再确认是否正确

这个跨应用操作链路非常低效。

我们的目标是把它压缩成：

1. 拍一下 / 对准一下 / 说一句
2. 系统提取动作意图
3. 系统给出可执行建议
4. 用户确认后直接完成动作

---

## 5. 产品定义

### 5.1 什么是 Visual-to-Tool OS

它不是一个完整手机系统，而是一个 **视觉到动作的系统中间层原型**。

它的职责是：

- 理解视觉信息
- 判断是否可执行
- 调用系统工具

所以它更像：

**AI 驱动的 OS 级动作路由层**

而不是：

- 单纯 OCR
- 单纯图片描述
- 单纯问答助手

### 5.2 核心理念

我们不追求“绝对完美的单次识别”。

我们追求的是：

- 结构化输出
- 容错机制
- 风险拦截
- 人在回路

也就是说：

> 即使识别不完美，系统依然应该稳定、可解释、可修复、可控。

---

## 6. 系统整体架构

系统划分为三层：

### 6.1 多模态感知层 Perception Layer

输入包括：

- 相机预览帧
- 单次拍照
- 语音识别文本
- 手动文字输入

目标：

- 把现实世界输入统一转成可供模型理解的多模态数据
- 后续支持端侧 OCR 文本框定位和质量预检

当前工程已具备：

- 相机预览与拍照
- 语音输入
- 手动文本输入

后续可增强：

- ML Kit / TFLite 文本框检测
- 清晰度、曝光度、目标区域评分
- 连续帧自动选帧

### 6.2 大模型决策与意图路由层 LLM Routing Layer

当前对接：

- 接口：`https://api-ai.vivo.com.cn/v1/chat/completions`
- 模型：`Volc-DeepSeek-V3.2`

模型不是用来自由发挥的，而是被约束为：

- 输出严格 JSON
- 自报置信度
- 输出结构化动作意图
- 在不确定时给出澄清追问话术

### 6.3 执行与交互层 Execution & UI Layer

客户端负责：

- 字段校验
- 融合置信度计算
- 风险阈值判断
- 确认卡片交互
- Android Intent 映射执行

这层决定了项目是不是“工程系统”，而不只是“模型 Demo”。

## 7. 核心技术方案

### 7.1 结构化意图契约

模型必须输出如下结构：

```json
{
  "action": "create_event | navigate | send_sms | tts_feedback | clarification | unknown",
  "confidence": 0.85,
  "payload": {
    "title": "提取或生成的标题",
    "time": "严格 ISO 8601 时间",
    "location": "具体 POI 或建筑编号",
    "phone_number": "纯数字字符串",
    "description": "详细上下文",
    "answer": "适合播报的摘要"
  },
  "fallback_query": "如果 confidence 低于阈值时的追问话术",
  "target_found": true
}
```

这样设计的好处：

- 模型输出可被程序稳定解析
- 风险策略可配置
- 后续扩展多场景时不需要推倒重来

### 7.2 三段式安全执行链

整个执行流程采用：

**Extract -> Suggest -> Confirm**

再进入实际执行。

#### Extract

应用先获取结构化 JSON。

#### Suggest

客户端根据：

- `action`
- `confidence`
- 必填字段完整度
- 风险级别

做二次判断。

#### Confirm

若达到可执行条件，则展示确认卡片；
若未达到，则触发 `clarification` 并通过 TTS 追问。

这个机制的本质是：

> 不让模型直接指挥系统，而让客户端做执行前的最后把关。

### 7.3 融合置信度

我们没有完全信任模型自报的 `confidence`。

当前工程中已经实现：

- `modelConfidence`
- `fusedConfidence`

融合思路包括：

- 模型语义置信度
- payload 完整度
- 视觉质量置信度
- 时序稳定度

这使得系统更适合处理现实世界中的模糊输入。

### 7.4 风险引擎

当前系统按照动作风险分级：

- `send_sms`：高风险
- `create_event`：中风险
- `navigate`：中风险
- `tts_feedback`：低风险

对应策略：

- 高风险动作必须严格拦截，不允许自动发送
- 中风险动作需要确认后执行
- 低风险动作可直接 TTS 输出

这部分已经体现在代码中的：

- `RiskPolicyEngine.kt`
- `ActionValidator.kt`

### 7.5 人在回路

这是本项目成立的关键。

如果没有人在回路，这个系统会变成：

- 一次识别
- 一次误判
- 一次误执行

有了人在回路，系统可以：

- 解释自己看到了什么
- 解释自己打算做什么
- 在不确定时主动追问
- 在高风险时要求确认

这就是工业级工程和实验型 Demo 的分界线。

## 8. 典型场景落地

当前最适合展示的不是单一行业，而是四类高频视觉任务：

### 8.1 链路 A：海报 / 通知 -> `create_event`

示例场景：

- 讲座海报
- 考研宣讲
- 会议通知

问题：

- 时间和地点通常分散在不同版块
- 格式不统一
- 相对时间需要推断

流程：

1. 用户拍摄海报
2. 模型提取标题、时间、地点
3. 界面展示“新建日程”确认卡片
4. 用户确认后写入日历

### 8.2 链路 B：地点信息 -> `navigate`

示例场景：

- 校园楼栋牌
- 会议地址
- 导视信息

问题：

- 地址文本常常不适合直接复制到地图
- 地点可能带别名或模糊前缀

流程：

1. 用户拍摄地点信息
2. 模型抽取核心地点
3. 界面展示“即将导航至”卡片
4. 用户确认后调用地图

### 8.3 链路 C：联系人或设备铭牌 -> `send_sms`

示例场景：

- 设备维护电话
- 服务联系人
- 临时通知协同

问题：

- 错发短信成本很高
- 自动发送不可接受

流程：

1. 系统提取电话和正文
2. 强制进入短信草稿模式
3. 只打开短信界面，不自动发送

### 8.4 链路 D：长文本 -> `tts_feedback`

示例场景：

- 菜单
- 复习资料
- 长通知

问题：

- OCR 逐字朗读负担太重
- 用户真正需要的是摘要

流程：

1. 模型先阅读理解
2. 输出简化后的重点信息
3. 系统直接 TTS 播报

---

## 8.5 为什么这四条链路最适合比赛展示

它们有三个共同特点：

- 都是现实生活中高频出现的碎片化视觉信息
- 都能自然衔接到手机原生工具能力
- 都能体现“先理解，再执行”的系统思路

相比只做图片描述，这四条链路更容易让评委看到：

- 输入不是终点
- 模型不是终点
- 系统动作闭环才是终点

## 10. 技术亮点总结

如果要让评委快速记住，我们建议浓缩成三点：

### 10.1 结构化 Schema

模型必须输出严格 JSON，而不是自由文本。

这让系统可以：

- 稳定解析
- 做字段校验
- 做风控分级
- 做后续执行映射

### 10.2 风控中间层

系统不是“识别完成就执行”，而是必须经过：

- Suggest 判定
- 阈值检查
- 确认或追问

这让项目从“模型 Demo”变成“工程系统”。

### 10.3 融合置信度与时序稳定

我们没有完全相信模型自报分数，而是引入：

- payload 完整度
- 视觉质量
- 时序稳定度

这使系统更适合处理真实场景里的模糊输入。

## 11. 答辩策略

### 11.1 不把自己定义成 App

答辩时建议直接说：

> 我们不是在做一个识图工具 App，而是在做一个 Visual-to-Tool OS 原型，探索大模型如何作为手机系统与现实世界之间的中间件。

### 11.2 展示容错能力

不要只展示成功案例。

建议故意加入一个模糊案例，展示：

- 系统没有直接执行
- 系统给出较低置信度
- 系统弹出确认或发起追问

这能明显提升评委对工程可靠性的判断。

### 11.3 路径对比

建议在 PPT 上展示一张对比图：

传统路径：

- 看信息
- 记信息
- 找 App
- 手输
- 保存

我们的路径：

- 输入
- 卡片确认
- 直接执行

这能直观体现效率提升。

## 12. 可直接口述的汇报摘要

我们做的是一个移动端视觉-动作操作系统原型。它不是单纯识图，而是让手机把现实世界中的视觉信息转成结构化意图，再通过客户端风控判断是否可以安全执行，最终调用日历、地图、短信和 TTS 等系统能力。  

这个系统的关键创新在于三点：第一，模型输出是严格 Schema，而不是随意自然语言；第二，客户端引入了风险引擎和融合置信度，不让模型直接控制系统；第三，中高风险动作必须通过确认卡片或追问修复后才能执行。  

目前工程上已经完成主界面、结构化 JSON 解析、风险策略、确认交互和四类核心动作链路，因此今天下午可以稳定演示“海报入日历”“地点去导航”“长文本摘要播报”和“短信草稿拦截”这几条主线。

第四段：

“目前我们已经完成了初版原型，可演示四条主链路：海报入日历、地点去导航、短信草稿、长文本播报。它们都能体现一个核心价值，就是把现实世界中的视觉信息，安全地转化成手机里的系统动作，这也是我们认为最符合手机 AI 助手和系统 AI 体验方向的创新点。”
